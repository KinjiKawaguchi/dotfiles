# コーディングスタイル

## ファイル構成

- ドメイン（機能）ごとにディレクトリを分ける。型ごと（models/, views/ 等）ではなく機能ごと（orders/, users/ 等）に構成する。
- 1ファイル200〜400行、上限800行。超える場合は責務を分割する。

## 関数・メソッド

- 1関数50行以内。超える場合は意味のある単位に分割する。
- ネストは4段以内。早期リターンで条件分岐を浅く保つ。
- 副作用を最小化し、純粋関数を優先する。

## 命名

- 変数名・関数名は意図を明確に表現する。省略形より可読性を優先する。
- boolean変数は `is`, `has`, `should`, `can` で始める。

## エラー処理

- 外部入力（API、ユーザー入力）はシステム境界でバリデーションする。
- 内部コードやフレームワークの保証には余分なバリデーションを入れない。
- エラーメッセージに機密情報を含めない。

## イミュータビリティ

- オブジェクトや配列は直接変更せず、新しいインスタンスを生成する。
- 状態変更が必要な場合は明示的なAPIを通じて行う。
